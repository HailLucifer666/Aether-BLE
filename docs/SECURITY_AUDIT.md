# Security Audit — Phase 6: Security & Discovery

**Scope:** `identity.py`, `realm.py`, `beacon_auth.py`, `pairing.py`, `discovery.py`,
`messages.py` (`build_hello_message` only), `bridge.py` (`_on_advertisement` beacon
path). Reviewed against `docs/PRD.md`, `docs/ARCHITECTURE.md`, `PROTOCOL.md` §11,
and `aether-bridge/SECURITY.md`.

**Verdict: no CRITICAL or HIGH findings that block shipping this phase as scoped
(LAN-trust, single-realm, solo operator).** One HIGH-severity spec/implementation
divergence is flagged (grace window is documented but not wired up) — it does not
create a new vulnerability today (no rotation code path exists yet either), but it
must be fixed before `realm.rotate_key()` is ever exercised in production, or the
rotation feature will silently lock out every node that hasn't refreshed within one
beacon interval. Everything else is MEDIUM/LOW/informational, consistent with the
explicitly documented and honestly-scoped LAN-trust posture.

---

## Findings

**[HIGH] Grace-window key rotation is documented but not implemented in the live verification path** — `aether-bridge/realm.py:36-43`, `aether-bridge/bridge.py:113-129,155` — `PROTOCOL.md` §11.3 and `Realm.verify_key()`/`key_history` promise that "verification is accepted against any of the last N key versions" so a rotation doesn't instantly break beacons still using the previous key. In practice, `Bridge.__init__` sets `self.realm_key = load_or_create_realm().current_key` (a single `bytes` value, not a `Realm`), and `_on_advertisement` calls `verify_beacon(payload, self.realm_key, last_counter)` — a single fixed key, never any of `key_history`. `Realm.verify_key()` is dead code; nothing in `bridge.py`, `beacon_auth.py`, or the wiring between them ever consults `key_version`/`key_history`. If `realm.rotate_key()` is ever called (there's no caller yet, but the API and the invariant doc both promise this works), every beacon still broadcasting with the pre-rotation key would be rejected immediately as `hmac_mismatch` — a hard, mesh-wide lockout, not the documented graceful grace window. This isn't exploitable today only because nothing calls `rotate_key()` yet in production code; it becomes a functional/availability bug (not itself a confidentiality/integrity break) the moment rotation ships. **Fix:** either (a) have `Bridge` hold the `Realm` object (not just `current_key`) and have `verify_beacon` (or a wrapper) try `realm.current_key` first, then fall back through `key_history` versions within the grace window, updating a per-uid_hash "which version last matched" if useful for logging; or (b) if rotation truly isn't wired up this phase, update `SECURITY.md`/`PROTOCOL.md` §11.3 to say grace-window verification is not yet connected to the beacon path, so the next engineer doesn't assume it works. Given `SECURITY.md` treats "confirmed product decision, do not simplify away" language elsewhere, silently having an unwired invariant is exactly the kind of gap that doc is meant to prevent — recommend closing the loop before this phase is called done, even though it doesn't block current single-key operation.

**[MEDIUM] Key material files have no explicit permission hardening** — `aether-bridge/identity.py:66-71`, `aether-bridge/realm.py:90-92`, `aether-bridge/beacon_auth.py:126-128` — `path.write_text(...)` is called with no `os.chmod`/`os.open(..., mode=0o600)` step anywhere for `identity.json` (Ed25519 private key), `realm.json` (HMAC key), or `beacon_counter.json`. On POSIX systems the resulting permissions depend entirely on the process umask (commonly `022`, giving `644` — world-readable). `SECURITY.md` item 6 explicitly acknowledges "protected only by OS filesystem permissions" as the accepted posture for this phase, so this is a known, documented trade-off rather than an oversight — but "protected by filesystem permissions" is only true if something actually sets restrictive permissions, and nothing does. **Recommendation (not a blocker given the documented LAN-trust/single-user posture):** set `0o600` on `identity.json` and `realm.json` at creation time (e.g. `os.open(path, os.O_CREAT | os.O_WRONLY, 0o600)` before writing, or `path.chmod(0o600)` immediately after `write_text`). Cheap to add, meaningfully reduces blast radius on any shared/multi-user machine, and costs nothing on the solo-operator target environment. Flag as a follow-up, not a ship-blocker, since the doc already scopes this honestly.

**[LOW] `AETHER_MFG_ID = 0xFFFF` — confirmed non-issue, follow-up only.** `bridge.py:39`. This is the Bluetooth SIG's reserved test/internal-use Company Identifier, valid for indefinite hobbyist/internal use per SIG allocation rules, as the code comment and `SECURITY.md` item 7 both correctly note. It is not a security vulnerability — 0xFFFF being "reserved for testing" does not make the *payload* under it any less authenticated; the HMAC construction doesn't depend on the manufacturer ID being secret or unique. The only real-world risk is a different hobbyist device also using 0xFFFF for unrelated manufacturer data, which `beacon_auth.verify_beacon`'s magic-byte/version/HMAC checks already filter out safely (wrong length or bad magic just gets logged at debug and dropped — confirmed in `bridge.py:150-152` and `beacon_auth.py:83-91`). No fix required now; correctly identified by the build agent as a non-blocking judgment call. Revisit only if/when Aether registers a SIG Company ID.

**[LOW] `uid_hash_from_name` derives identity from the beacon's advertised name string, not a real per-device credential — confirmed placeholder, not a regression.** `beacon_auth.py:34-43`. `uid_hash = sha256(name)[:4]` means "identity" is really "whatever string the beacon advertises as its name," authenticated only insofar as the HMAC proves *someone holding the realm key* sent this reading, not that the specific claimed uid_hash belongs to a specific physical device. Practically: within a single realm, any paired member holding the realm key could construct a valid HMAC for *any* `uid_hash` value, including one matching another member's beacon name — there is no per-device signing key binding the HMAC to a specific sender identity (that's what Ed25519 node identity is for elsewhere in this phase, but it isn't used here). This is explicitly scoped as single-user/single-beacon for this phase per the PRD and `ARCHITECTURE.md` ("uid_hash is single-user for now"), and `election.py`/multi-user disambiguation is explicitly untouched. It is a correct, honestly-documented placeholder for this phase, not a hidden gap — but flag clearly for Phase 7+: if/when multi-beacon or multi-user support is built, uid_hash must be tied to something the realm-key holder can't freely forge on behalf of another member (e.g. a per-device sub-key or an Ed25519-signed claim), or any compromised/malicious paired member can impersonate any other member's beacon within the realm.

**[LOW / informational] `cryptography>=42.0` has no upper bound pin — dependency hygiene, not an active CVE.** `aether-bridge/requirements.txt`. `cryptography` 42.0.0–42.0.3 predate the fix for CVE-2024-26130 (NULL-pointer dereference via crafted PKCS12 data, unrelated to this project's Ed25519/HMAC-only usage but still worth tracking); later CVEs (e.g. CVE-2024-12797, fixed in 44.0.1, involving raw private key deserialization paths) postdate 42.0 entirely. Because the constraint is `>=42.0` with no ceiling, a fresh `pip install` today resolves to whatever the latest published version is (almost certainly already patched) — so there is no live exposure right now, but the requirements file offers no protection against a *future* regression and doesn't pin to a known-good floor above the CVE-fixed versions. Same applies to `zeroconf>=0.130` and `qrcode>=7.4` — no known CVEs found against either at time of review, both are lower-risk pure-Python libraries without CVE history matching this project's usage (no XML parsing, no unsafe deserialization in the API surface actually used). **Recommendation:** bump the floor to `cryptography>=44.0.1` (or whatever the current patched minimum is at merge time) so a clean install can't land on a pre-patch version, and run `pip list --outdated` / `pip-audit` periodically per the project's own dependency-security practice. Not a blocker; routine hygiene.

## Verified as sound (no issues found)

- **HMAC construction and comparison** (`beacon_auth.py:46-49,100-102`). Message is `uid_hash(4B) || counter(4B)` under `HMAC-SHA256(realm_key, ...)`, truncated to 8 bytes — matches `PROTOCOL.md` §11.2 exactly. Comparison uses `hmac.compare_digest(received_mac, expected_mac)`, not `==` — constant-time as required. No manual byte-loop comparison anywhere in the file.
- **Replay/counter logic** (`beacon_auth.py:97-98`). Rejection condition is `counter <= last_counter` (not `<`), correctly rejecting an exact-replay of the same counter value, matching both `SECURITY.md` invariant #1 and `PROTOCOL.md` §11.2's explicit "equal counters are rejected, not just lower ones" language. Verified this is checked before the HMAC comparison but does not create an exploitable timing oracle for this threat model (both checks are cheap/constant-ish relative to BLE scan timing granularity; there's no secret being leaked by which check fails first since `reason` strings are the only observable output and they're only logged locally, never transmitted).
- **Counter persistence, single-process race safety** (`beacon_auth.py:107-128`, `bridge.py:147-163`). `BeaconCounterStore` loads eagerly at construction and rewrites the whole file on every `set_last_counter` call. `_on_advertisement` is bleak's detection callback, invoked synchronously on the single asyncio event loop thread — there is no concurrent writer within one `Bridge` process, so no read-check-write TOCTOU race in the deployed topology (one bridge process per scanner, per `ARCHITECTURE.md`/`PROTOCOL.md` §2 topology: scanners never share a counter store). If a future change ever ran two `Bridge` instances against the same counter file this would need a lock or atomic replace — not applicable to the current single-scanner-per-process design, flagging only as a latent assumption worth a code comment.
- **Realm key never logged** (`bridge.py:156-160`). Confirmed `logger.debug("dropped beacon advertisement: %s", result.reason)` only ever logs the short enum-like `reason` string (`"bad_length"`, `"bad_magic"`, `"bad_version"`, `"replay"`, `"hmac_mismatch"`), never the payload bytes, the realm key, or the computed/received MAC. Matches `SECURITY.md` invariant #4 exactly.
- **Ed25519 usage** (`identity.py`, `pairing.py`). Standard `cryptography.hazmat.primitives.asymmetric.ed25519` calls only — key generation via `Ed25519PrivateKey.generate()` (CSPRNG-backed), no custom signature verification logic present in the reviewed files (note: `messages.py`'s `build_hello_message` only *builds* the `sig` field as an opaque hex string parameter — it does not itself sign or verify; the actual signing/verification call sites for `hello` were not present in the files in scope and should be confirmed elsewhere before Phase 6 is considered fully closed, since an unsigned or unverified `sig` field would make the `hello` handshake authentication decorative).
- **Realm key generation** (`realm.py:83`). `os.urandom(32)` — correct CSPRNG source for a 256-bit HMAC key, not `random`/`hashlib` seeded from anything predictable.
- **Grace window bound** (`realm.py:23,53-56`). `GRACE_WINDOW_VERSIONS = 3` is a finite constant, and `rotate_key()` prunes `key_history` entries below `key_version - GRACE_WINDOW_VERSIONS + 1` on every rotation — an old key cannot remain accepted indefinitely, matching `SECURITY.md` invariant #5. (Caveat: as noted in the HIGH finding above, this pruning logic is currently unreachable from the live beacon-verification path since nothing calls `rotate_key()` or consults `key_history` yet — the bound itself is correctly implemented, just not yet wired to anything that exercises it.)
- **QR pairing ceremony network exposure** (`pairing.py:58-79`). `PairingCeremony.__init__` defaults `host="127.0.0.1"` and `asyncio.start_server(self._handle_peer, self.host, self._requested_port)` binds to exactly that host — **loopback only by default, not `0.0.0.0`**. As written, the listener does not accept connections from other hosts on the LAN; a peer must already have loopback access to this machine to connect, which contradicts the "scan a QR and dial in from a phone" user story in `PRD.md` story 2 (a phone on the LAN cannot reach `127.0.0.1` on the PC). This looks like it will need to bind to a LAN-reachable interface address (`0.0.0.0` or the actual LAN IP) for the real cross-device pairing flow described in the PRD to function at all — but that is a **functional gap for a later integration** (the module's own docstring says "this module only needs to make the offering side work... testable by driving the socket directly," and the scanning/Android side is explicitly out of scope this phase), not a security regression in what's shipped today. Flagging as a correctness/completeness note for whoever wires up the real cross-device pairing flow: when the bind address is widened to be LAN-reachable, revisit finding below on invite-token exposure.
- **Pairing ceremony key-exchange integrity during the pairing window** (`pairing.py:86-107`). Within the *current* loopback-only scope, there is no LAN-interception risk since no LAN traffic occurs. If/when the listener is bound to a LAN address for real cross-device use (see above), note for that follow-up: the exchange is plaintext TCP (no TLS), and admission is gated purely by `token == self.realm_invite_token` — a plaintext bearer token compared with Python's `!=` (`pairing.py:93`), not `hmac.compare_digest`. For a short-lived, one-shot, operator-initiated local pairing window this is a low-severity timing side channel (the token is single-use, ceremony is torn down after, and exploiting a timing leak over a live network in the few-second pairing window is impractical) — but recommend switching to `hmac.compare_digest(token.encode(), self.realm_invite_token.encode())` for defense-in-depth once this listener is LAN-reachable, consistent with the project's own stated principle of never hand-rolling comparison of secret-derived values. Also worth deciding at that point whether the invite token should be single-use/expiring (currently it's a static string compared on every connection attempt for the ceremony's lifetime — fine for a manual, attended, short-lived pairing window, but worth a comment noting that assumption explicitly).
- **LAN-trust scoping honesty** (`PROTOCOL.md` §11.4, `SECURITY.md` "What is explicitly NOT protected against"). Confirmed no code path in the reviewed files silently assumes transport confidentiality it doesn't have. `bridge.py`'s WebSocket server (`websockets.serve`, `bridge.py:303`) is plain `ws://`, matching the documented deferral. `pairing.py`'s TCP listener is also plaintext, again consistent with the documented scope (loopback-only today, so no LAN exposure to misrepresent). No file references TLS/`wss://`/Noise anywhere as if it were active. The security annex's claims and the code's actual behavior line up.
- **No hardcoded secrets.** Swept all reviewed files for API keys, passwords, and hardcoded tokens (`grep` for common secret patterns) — none found. `REALM_KEY` and similar in test files are clearly test-only fixtures (`test_bridge_beacon_auth.py:24`, marked and scoped to `tests/`).
- **mDNS discovery** (`discovery.py`). No credential material is exposed via mDNS TXT records — only `node_id` (a public fingerprint, not a secret) is published. `zeroconf` usage is standard registration/browse; no injection surface identified (service name/instance name are locally-controlled operator input, not attacker-controlled remote input, in this phase's topology).

**Overall:** Phase 6 as implemented is consistent with its own documented threat model. The core cryptographic invariants the build agent's `SECURITY.md` calls out (constant-time HMAC compare, strict `<=` replay rejection, counter persistence, no key material in logs, finite grace window) are all correctly implemented in the code, not just asserted in the doc. The one HIGH finding is a completeness gap (grace window promised but not connected) rather than an active vulnerability, since the rotation feature it protects has no caller yet — but it should be closed or explicitly re-scoped before rotation ships, so a future "just call `rotate_key()`" change doesn't silently break the mesh. Nothing found contradicts the explicit LAN-trust/deferred-transport-encryption stance; the code is honest about what it does and doesn't protect.
